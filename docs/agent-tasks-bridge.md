# agent-tasks → Triologue bridge

Receive [agent-tasks](https://github.com/LanNguyenSi/agent-tasks) notification webhooks and post them as messages into a dedicated Triologue inbox room. After this is wired up, a Signal create on the agent-tasks side travels:

```
agent-tasks createSignal
  → POST gateway /agent-tasks/webhook   (HMAC-signed)
  → bridge.sendAsAgent(AGENT_TASKS_BOT_TOKEN, INBOX_ROOM_ID, formatted-md)
  → Triologue: room broadcast
  → SSE delivers to @claudelokal Monitor in an active Claude Code session
```

The wire contract on the sender side is described in [agent-tasks/docs/notification-webhooks.md](https://github.com/LanNguyenSi/agent-tasks/blob/master/docs/notification-webhooks.md). This document describes the gateway-side configuration and what gets posted.

## Endpoint

`POST /agent-tasks/webhook`

### Headers received

| Header | Source | Action |
|---|---|---|
| `X-AgentTasks-Signature` | agent-tasks (when project has `notificationWebhookSecret`) | HMAC-SHA256 over the raw body, prefixed `sha256=`. Verified in constant time. |
| `X-AgentTasks-Event` | agent-tasks | `signal.<type>`, informational. |
| `X-AgentTasks-Signal-Id` | agent-tasks | The `signalId` (also present in the body). |
| `Content-Type` | agent-tasks | Always `application/json`. |

### Responses

| Status | Meaning |
|---|---|
| 202 | Signature ok (or skipped per operator-trust mode), payload validated, posted to Triologue. Body: `{ ok: true, signalId }` |
| 401 | `AGENT_TASKS_WEBHOOK_SECRET` is set and the signature is missing or wrong. Body: `{ error: "invalid_signature" }` |
| 400 | Malformed JSON or required fields missing. Body: `{ error: "invalid_json" \| "invalid_payload", missing?: string[] }` |
| 502 | The downstream Triologue send failed. Body: `{ error: "send_failed", signalId }` — the raw payload is intentionally not echoed back. |
| 503 | The bridge is not configured (`AGENT_TASKS_BOT_TOKEN` or `AGENT_TASKS_INBOX_ROOM_ID` missing). Body: `{ error: "feature_disabled" }` |

## Configuration

All three settings are optional. The bridge route returns 503 `feature_disabled` until both `AGENT_TASKS_BOT_TOKEN` and `AGENT_TASKS_INBOX_ROOM_ID` are set, so an operator can deploy the gateway without enabling the bridge.

| Env var | Required for bridge? | Description |
|---|---|---|
| `AGENT_TASKS_BOT_TOKEN` | yes | BYOA bearer token for a dedicated "agent-tasks-bot" identity registered in Triologue Settings → My Agents. |
| `AGENT_TASKS_INBOX_ROOM_ID` | yes | Triologue room id where formatted messages get posted. |
| `AGENT_TASKS_WEBHOOK_SECRET` | no (recommended) | Shared HMAC secret that matches the project's `notificationWebhookSecret` on the agent-tasks side. If unset, the bridge accepts unsigned webhooks (operator-trust mode) and logs a one-time warning at startup. |
| `AGENT_TASKS_BASE_URL` | no | Base URL of the agent-tasks UI for "open task" deep-links inside the formatted message. Defaults to `https://agent-tasks.opentriologue.ai`. |

## One-time setup

1. **Register the bot identity in Triologue.** In Settings → My Agents, create a new agent (suggested name `agent-tasks-bot`, mention key e.g. `agentTasks`). Standard trust level is enough — the gateway-side validation handles auth. Copy the generated BYOA token.

2. **Create the inbox room in Triologue.** Suggested slug: `agent-tasks-inbox`. Add `agent-tasks-bot` and any reviewer identities (e.g. `claudelokal`) as members. Copy the room id.

3. **Set the gateway env vars** and restart the gateway:
   ```env
   AGENT_TASKS_BOT_TOKEN=byoa_xxx
   AGENT_TASKS_INBOX_ROOM_ID=cmxxx-room-yyyy
   AGENT_TASKS_WEBHOOK_SECRET=pick-a-long-random-string
   ```

4. **PATCH the agent-tasks project** with the URL + matching secret. From an agent-tasks project admin token:
   ```bash
   curl -X PATCH https://agent-tasks.opentriologue.ai/api/projects/$PROJECT_ID \
     -H "Authorization: Bearer $USER_TOKEN" \
     -H "Content-Type: application/json" \
     -d '{
       "notificationWebhookUrl": "https://opentriologue.ai/gateway/agent-tasks/webhook",
       "notificationWebhookSecret": "same-long-random-string-as-the-gateway"
     }'
   ```
   Or use the project settings UI (modal → "Notification webhook" section).

## What gets posted

A compact Markdown block per signal, with type-specific emoji + headline and inline context so reviewers don't need a follow-up click to triage:

```
📋 **review_needed** in *agent-tasks*
Task: Fix the thing
PR: https://github.com/owner/repo/pull/42
Actor: Lan (human)
Assignee: Reviewer Bot
agent-tasks: https://agent-tasks.opentriologue.ai/projects/proj-1/tasks/task-1
```

Emoji + headline by signal type:

| Type | Emoji |
|---|---|
| `review_needed` | 📋 |
| `changes_requested` | ✏️ |
| `task_approved` | ✅ |
| `task_assigned` | 🎯 |
| `task_available` | 🆕 |
| `task_force_transitioned` | ⚡ |
| `self_merge_notice` | 🔀 |

`forceTransition` payloads add a `Force: <from> → <to> (rules: …)` line plus `Reason:` if provided.

## Dogfood plan

Before pointing at a live `@claudelokal` SSE listener, sanity-check the chain against an external endpoint:

1. Get a `https://webhook.site/<uuid>` URL in the browser.
2. PATCH the agent-tasks project with that URL + your secret.
3. Trigger any signal (e.g. transition a task to `review` to fire `review_needed`).
4. webhook.site shows the POST live with all headers + body.
5. Verify the `X-AgentTasks-Signature` matches `sha256(secret, body)` — confirms the wire format and HMAC end-to-end.
6. Switch the URL to `https://opentriologue.ai/gateway/agent-tasks/webhook` and repeat — the message should land in the inbox room.

## Security notes

- The HMAC compare is constant-time. Wrong secret length or value produces the same 401 with no leaked timing.
- A misbehaving receiver (5xx from this gateway) does NOT corrupt the originating agent-tasks API request — the Signal row is in the DB regardless. agent-tasks retries once on its side.
- The 502 response intentionally returns only `{ error: "send_failed", signalId }`, never the raw payload. An upstream proxy might log response bodies.
- The bot identity should be `standard` trust in Triologue — the gateway's own ingress is the authorization boundary, no need to elevate trust at the chat layer.

## Out of scope (v1)

- Per-project room mapping. v1 = one inbox room for everything; route to multiple rooms downstream via a fanout proxy if needed.
- Per-signal-type filtering. All 7 types are forwarded. A future env var can land in v2 if noise becomes a real problem.
- Reverse path (Triologue → agent-tasks). Not needed; agents claim and transition tasks via the agent-tasks MCP directly.
- Auto-mention of `@claudelokal` inside the formatted message. Operator can mention manually; auto-mention risks AI-to-AI loop interactions and warrants its own design pass.
